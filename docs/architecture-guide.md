# Architecture Guide

This document explains the directory layout and the role of each file within the `validation-framework/`.

---

## 📂 Directory Layout

```text
validation-framework/
├── config/
│   ├── page-validation.config.ts  # Interfaces, defaults, and validator toggling
│   ├── urls.json                  # Manual URL fallback list
│   └── sitemap-urls.json          # Cached sitemap paths (auto-generated at runtime)
├── docs/
│   ├── architecture-guide.md      # This file
│   ├── functionality-guide.md     # Usage scenarios, execution flow, and extension guide
│   └── test-specifications.md     # Detailed per-validator assertion specifications
├── pages/
│   └── HomePage.ts                # Page Object Model (POM) encapsulating selectors
├── services/
│   └── page-validation.service.ts # Central registry and validation orchestrator
├── tests/
│   ├── global-setup.ts            # Sitemap pre-fetcher and port coordinator
│   ├── mock-server.js             # Local HTTP server for offline framework testing
│   └── page-validation.spec.ts    # Playwright E2E spec runner
├── utils/
│   └── network.ts                 # HTTP helpers, URL validation, and sitemap XML parser
└── validations/
    ├── page-health.validator.ts   # HTTP status and redirect destination validation
    ├── seo.validator.ts           # HTML head metadata tag validations
    ├── header.validator.ts        # Logo, navigation, mega-menu, and search validations
    └── footer.validator.ts        # Footer sections, social links, locale, and copyright validations
```

---

## 🏗️ Layer & File Reference

### 1. Configuration Layer (`config/`)

Defines parameters and type contracts for all validation targets.

- **[page-validation.config.ts](../config/page-validation.config.ts)**
  Defines TypeScript interfaces (`SeoValidationConfig`, `HeaderValidationConfig`, `FooterValidationConfig`) and exports `defaultConfig` with global default selectors, tag expectations, and validator flags. Also reads and parses `ACTIVE_VALIDATORS` from the environment.

- **[urls.json](../config/urls.json)**
  An array of relative page paths (e.g., `["/", "/about"]`) used when `USE_SITEMAP=false`.

- **[sitemap-urls.json](../config/sitemap-urls.json)**
  Auto-generated by `global-setup.ts` before tests begin. Contains absolute URLs extracted from the target sitemap.

---

### 2. Page Object Model Layer (`pages/`)

Decouples DOM structure from validation logic.

- **[HomePage.ts](../pages/HomePage.ts)**
  Exposes typed Playwright `Locator` getters for header, footer, logo image, search modal, copyright section, and locale dropdown. Resolves selectors from `defaultConfig` to keep validators selector-agnostic.

---

### 3. Orchestration & Service Layer (`services/`)

Controls execution and validator registration.

- **[page-validation.service.ts](../services/page-validation.service.ts)**
  - Maintains `VALIDATOR_REGISTRY` — a map of validator type keys to their runner implementations.
  - Reads `activeValidators` from `defaultConfig` (driven by `ACTIVE_VALIDATORS` env var) to determine which checks to run.
  - Exposes `validatePage()` (runs all enabled validators in sequence) and individual `validateHealthOnly`, `validateSeoOnly`, `validateHeaderOnly`, `validateFooterOnly` methods.
  - Automatically skips content validators (`seo`, `header`, `footer`) for redirect pages.

---

### 4. Spec Runner Layer (`tests/`)

Drives compilation and test execution.

- **[global-setup.ts](../tests/global-setup.ts)**
  Runs before any test file compiles. If `USE_SITEMAP=true`, it spins up a temporary fork of the local mock server (if testing locally), fetches and parses the sitemap XML, writes URL paths to `config/sitemap-urls.json`, then shuts the fork down.

- **[page-validation.spec.ts](../tests/page-validation.spec.ts)**
  Synchronously resolves the URL list (from sitemap cache or manual file) and registers one `test()` block per URL. Each test delegates to `PageValidationService.validatePage()` and wraps each validator step in a `test.step()` with `expect.soft` to ensure all validators run even when earlier ones fail.

- **[mock-server.js](../tests/mock-server.js)**
  A zero-dependency Node.js HTTP server. Serves dummy HTML pages, canonical links, social icons, redirect responses, and a mock `sitemap.xml`. Allows full offline testing without hitting any remote environment.

---

### 5. Utility Layer (`utils/`)

Foundational networking helpers used across validators.

- **[network.ts](../utils/network.ts)**
  - `isValidUrl(url)` — validates URL syntax using the native `URL` constructor.
  - `verifyHttpStatus(requestContext, url)` — makes a raw, lightweight HTTP request via Playwright's `APIRequestContext` without opening a browser tab; used to verify redirect status codes and logo image availability.
  - `parseSitemapUrls(xml)` — extracts absolute URL strings from an XML sitemap payload using RegExp.

---

### 6. Validation Layer (`validations/`)

Houses the concrete assertion libraries for each check category.

- **[page-health.validator.ts](../validations/page-health.validator.ts)**
  Confirms correct HTTP status codes. For redirect pages (3xx), performs a raw HTTP request without following redirects to inspect the `Location` header. For standard pages (200), uses `page.goto()` and checks browser response status.

- **[seo.validator.ts](../validations/seo.validator.ts)**
  Asserts `<title>`, `<meta name="description">`, `<link rel="canonical">`, `<meta name="robots">`, Open Graph (`og:*`), and Twitter Card (`twitter:*`) tags are present and non-empty.

- **[header.validator.ts](../validations/header.validator.ts)**
  Verifies logo visibility and image load success, navigation container presence, mega-menu drawer trigger interaction, and search input keyboard focus.

- **[footer.validator.ts](../validations/footer.validator.ts)**
  Verifies footer section count, social link href syntax and domains, language selector dropdown interaction and `lang` attribute update, copyright year text, and App Store / Play Store link validity.
